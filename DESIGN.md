# 민원 답변 멀티에이전트 × 픽셀 에이전트 결합 설계 문서

> AI·데이터 분석 전문인재 양성과정 6차수 과제
> 민원 답변 멀티에이전트 서비스의 동작을 픽셀 오피스로 실시간 시각화하는 결합 프로젝트

---

## 1. 과제 목표

OpenRouter 기반 **민원 답변 멀티에이전트 서비스**(`minwon_multiagent.html`)를 실행했을 때,
6개 AI 에이전트의 동작이 **픽셀 오피스(pixel-agents)** 안에서 한 명씩의 캐릭터로 시각화되도록 두 시스템을 결합한다.

- 각 에이전트 = 사무실의 한 직원(캐릭터)
- 평소에는 자리에 앉지 않고 **사무실을 배회하거나 소파에서 대기**
- 자신이 일할 차례가 되면 **자기 책상으로 걸어가 업무(읽기/작성/검수) 동작**을 수행
- 캐릭터 **머리 위에 역할명 라벨**을 달아 누가 어떤 에이전트인지 식별 가능

---

## 2. 시스템 구성

### 2.1 민원 서비스 (클라이언트)

- 단일 HTML 파일. 브라우저에서 실행.
- `민원 담당자` 화면에서 민원 입력 → `답변 초안 생성` 클릭 → 6개 에이전트가 멀티에이전트 방식으로 동작.
- 동작 시점마다 픽셀 오피스로 **hook 이벤트를 POST**하는 코드(`emitPixel`)가 이미 내장되어 있음.
- 전송 대상: `http://127.0.0.1:3100/api/hooks/minwon`

### 2.2 픽셀 오피스 (서버 + 시각화)

- `pixel-agents` 오픈소스. VS Code 확장이자, `npx pixel-agents`로 실행 가능한 **standalone HTTP 서버 + 브라우저 SPA**.
- 포트 **3100**, `127.0.0.1` 바인딩.
- `POST /api/hooks/:providerId` 라우트로 hook 이벤트를 수신해 캐릭터 상태로 변환.
- Read → 읽기 애니메이션, Write/Edit → 타이핑 애니메이션.

---

## 3. 6개 에이전트 매핑

| key | 역할(머리 위 라벨) | 처리 단계 | 동작(tool) | 애니메이션 |
|------|---------------------|-----------|------------|-------------|
| `orch` | 조정자/분류 | STEP1 (병렬) | Read | 읽기 |
| `law` | 법령·근거 조사관 | STEP1 (병렬) | Read | 읽기 |
| `prec` | 소관·선례 조사관 | STEP1 (병렬) | Read | 읽기 |
| `tone` | 톤·리스크 검토관 | STEP1 (병렬) | Read | 읽기 |
| `draft` | 답변 작성가 | STEP2 | Write | 타이핑 |
| `crit` | 품질 검수관 | STEP3 (검수 루프) | Edit | 타이핑 |

**처리 흐름**
1. **STEP1 — 병렬 팬아웃**: `orch·law·prec·tone` 4명이 동시에 자기 책상으로 가서 읽기 동작.
2. **STEP2 — 통합 작성**: `draft`가 책상으로 가서 타이핑(초안 작성).
3. **STEP3 — 검수 루프**: `crit`가 책상으로 가서 검수. 미달 시 `draft`로 되돌려 재작성(최대 2회).

## 4. 연동 프로토콜 (hook 이벤트)

민원 서비스가 픽셀 오피스로 보내는 이벤트는 Claude Code hook 규격을 따른다.

| 이벤트 | 시점 | 주요 필드 | 픽셀 오피스 동작 |
|--------|------|-----------|------------------|
| `SessionStart` | 처리 시작 | `cwd` = 역할명 | 외부 세션 staging (역할 정체성 부여) |
| `PreToolUse` | 에이전트 작업 시작 | `tool_name`(Read/Write/Edit), `tool_input.status` | 캐릭터 등장·활성화 → 책상으로 이동해 동작 |
| `PostToolUse` | 작업 완료 | — | 동작 종료 |
| `Stop` | 에이전트 종료 | — | 대기 상태 복귀 |

```
민원 HTML  ──POST /api/hooks/minwon──▶  pixel-agents 서버
  emitPixel()                              hookEventHandler
                                              │
                                              ▼
                                        캐릭터 상태 머신
                                     (idle/배회 → walk → type/read)
                                              │
                                              ▼
                                       브라우저 픽셀 오피스
```

## 5. 결합 시 메워야 할 갭과 해결 방안

### 5.1 [핵심] 인증 토큰 갭 → **localhost 무인증 허용으로 해소**

- 픽셀 오피스의 hook 라우트에는 `Authorization: Bearer <token>` 인증이 항상 걸려 있다. 토큰은 서버 기동 시 무작위 생성되어 `~/.pixel-agents/server.json`에 저장된다.
- 민원 HTML의 `emitPixel`은 Authorization 헤더를 보내지 않으며, 브라우저는 그 파일을 읽을 수도 없다 → 모든 hook이 **401로 폐기**된다.
- **해결**: `server/src/httpServer.ts`의 hook 라우트 인증(`bearerAuth`)을 수정하여, **`127.0.0.1`/`::1`에서 들어온 요청은 토큰 없이 통과**시킨다.
  - WebSocket 라우트는 이미 standalone(비embedded) 모드에서 localhost 무인증을 허용하고 있어, 같은 정책을 hook 라우트에 일관되게 적용하는 형태가 된다.
  - 외부 네트워크 노출은 서버가 `127.0.0.1`에만 바인딩되므로 발생하지 않는다.

### 5.2 provider 라우팅

- URL의 `minwon`은 라우트 검증 패턴(`^[a-z0-9-]+$`)을 통과하며, 내부 핸들러는 providerId를 사실상 무시하고 claude provider 로직으로 처리한다 → **provider 신규 등록 불필요.**

### 5.3 역할 라벨(머리 위 표시)

- 캐릭터의 정체성은 `SessionStart`의 `cwd`(역할명)에서 온다.
- 캐릭터 **머리 위에 역할명 라벨이 항상 보이도록** webview 렌더링을 보강한다.

### 5.4 초기 대기 동작 (배회/소파) → 차례에 책상으로

- 6명은 처음부터 책상에 배정하지 않는다. **초기 상태: 사무실 배회 또는 소파 대기.**
- 해당 에이전트의 `PreToolUse`가 도착하면 그때 **자기 책상으로 걸어가 동작**, `Stop` 후 다시 대기(배회/소파)로 복귀.
- 책상은 6개 이상 배치하여 동시(STEP1 병렬) 작업 시 자리가 겹치지 않게 한다.

## 7. 실행·검증 절차 (요약)

1. `pixel-agents` 클론 → 의존성 설치 → 빌드 → `npx pixel-agents`(포트 3100) 기동.
2. 브라우저로 `http://127.0.0.1:3100` 접속해 픽셀 오피스 표시 확인.
3. `minwon_multiagent.html`을 브라우저로 열고 개발자 설정에서 OpenRouter 키 입력, 픽셀 연동 ON·서버주소 `http://127.0.0.1:3100` 확인.
4. 민원 입력 후 `답변 초안 생성` → 픽셀 오피스에서 6명이 차례대로 책상으로 가 동작하는지 확인.
5. `/api/health`와 hook 수신 로그로 401 없이 이벤트가 들어오는지 검증.
